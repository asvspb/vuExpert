# tdd-loop-v1

> Пак best practices: тесты как спецификация, цикл generate-test-fix. Версия v1.

## Принцип

> **Тесты — это спецификация.** Если поведение не описано тестом, его нет. ИИ-агент, у которого нет tight feedback loop, «уверенно говорит, что всё ок», ломая молча.

## TDD-петля с ИИ

Классический цикл, адаптированный под агентов:

```
1. КРАСНЫЙ: написать тест на требование (REQ) → он падает (фичи ещё нет)
2. ЗЕЛЁНЫЙ: написать минимальный код → тест проходит
3. РЕФАКТОРИНГ: улучшить код → тесты всё ещё зелёные
4. ПОВТОР для следующего REQ
```

**С ИИ-агентом:**
1. Агент читает REQ из `requirements.md`.
2. Генерирует тест первым (по EARS-формулировке).
3. Запускает — падает (красный).
4. Генерирует реализацию.
5. Запускает — должен позеленеть.
6. Если красный → итерирует (см. `verify/VERIFY.md`).

## Покрытие

**Минимум:**
- Unit-тесты на бизнес-логику (сервисный слой).
- Интеграционные на ключевые эндпоинты (CRUD, auth).
- Smoke-тест на запуск приложения.

**Идеал:** каждое REQ/NFR покрыто хотя бы одним тестом (см. трассировку в requirements.md).

## Что тестировать

| Слой | Что | Инструмент |
|---|---|---|
| Бизнес-логика | сервисы, чистые функции | Vitest / pytest |
| API | эндпоинты (запрос→ответ) | supertest / TestClient (FastAPI) |
| БД | модели, миграции | интеграционные с test-БД |
| UI (fullstack) | компоненты, поведение | Testing Library + Vitest |
| E2E | ключевые сценарии | Playwright |

## Антипаттерны

- ❌ Тесты после кода «для галочки» — они тестируют реализацию, а не поведение.
- ❌ Тесты, которые ничего не проверяют (`expect(true).toBe(true)`).
- ❌ Тесты с внешними зависимостями (сеть, прод-БД) без моков.
- ❌ «Работает у меня» без воспроизводимого теста.
- ❌ Помечать задачу done, если тесты красные.

## Связь с верификацией

Цикл generate-test-fix (`verify/VERIFY.md`) — это TDD-петля на уровне всего проекта: после генерации запускается полный набор, ошибки кормятся обратно агенту.
