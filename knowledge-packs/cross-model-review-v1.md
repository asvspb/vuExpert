# cross-model-review-v1

> Пак best practices: разделение generation и verification. Версия v1.

## Принцип

> **Код пишет одна модель, ревьюит — другая (или человек).** AI-код внешне убедителен и скрывает дефекты — нужен внешний взгляд.

Это ключевая дисциплина: ИИ-агент, который сам себя ревьюит, склонен подтверждать собственные ошибки. Другая модель ловит слепые зоны.

## Процедура

1. **PR создан** (код сгенерирован агентом А).
2. **Запусти review-промпт в другой модели/агенте** (агент Б) или попроси человека.
3. **Фокус ревью:**
   - Соответствие конвенциям репозитория (AGENTS.md).
   - Скрытые дефекты (edge cases, race conditions, SQL-injection).
   - Дублирование логики (агент мог переизобрести существующее).
   - Тесты покрывают поведение, а не реализацию.
   - Нет hardcoded секретов / магических чисел.
4. **Merge только после «looks good»** от ревьюера.

## Review-промпт (шаблон для агента-ревьюера)

```
Отревьюй этот PR. Контекст проекта — в AGENTS.md.
Проверь:
1. Соответствует ли код конвенциям AGENTS.md?
2. Есть ли скрытые дефекты или edge cases?
3. Дублирует ли код уже существующую логику в репо?
4. Покрывают ли тесты поведение (а не реализацию)?
5. Нет ли захардкоженных секретов/параметров?
Дай конкретный список замечаний со ссылками на файлы/строки.
Не пиши «всё хорошо» — найди минимум 2 пункта для улучшения.
```

## Комбинации моделей

| Генерация | Ревью | Комментарий |
|---|---|---|
| Локальная (GLM/Phi) | Облачная (Claude/GPT) | Локальная слабее — внешний ревью критичен |
| Claude | Gemini | Разные слепые зоны |
| Любая ИИ | Человек | Лучший вариант для критичного кода |

## Когда обязательно человек

- Код, связный с безопасностью (auth, платежи).
- Меняет архитектуру (новые слои, паттерны).
- Бизнес-критичная логика.
- Deployment-скрипты, миграции БД.

## Антипаттерны

- ❌ Одна модель пишет и ревьюит (подтверждает себя).
- ❌ «Looks good to me» без конкретики — требуй минимум 2 замечания.
- ❌ Merge без ревью «потому что срочно».
- ❌ Ревью только на стиль (форматирование) — пропуская логику.
